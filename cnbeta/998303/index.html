牺牲稳定换更短发布周期 Firefox的这步棋真的下对了吗？

July 02, 2020

去年九月，Mozilla 宣布了让很多人感到兴奋的消息– Firefox 浏览器的升级周期从此前的六周时间缩短至四周。对于用户来说，这意味着用户能够更快获得稳定版本。

在当时发布的公告中，Mozilla 承诺最终构建的稳定版会和此前一样经过严苛的测试。公司表示：“更短的发布周期提供了更大的灵活性，以支持产品规划和因业务或市场需求而产生的优先级变化。周期缩短至四周，在提供高质量的应用程序以及相同严苛的稳定测试前提下让我们更敏捷、更快速的推出新功能。”

三月初上线的 Firefox 74 是首个接受新周期的版本，之后版本将会每隔四周进行更新。不过伴随着 Mozilla 按下快进键，缩短周期的一些不足也逐渐显现出来。

虽然该公司确实已经成功地接受了这个四周的发布时间表，但其承诺的严格测试却不是这样的。Firefox 76/77/78 三个重要版本均出现了各种严重的问题，导致 Mozilla 被迫暂停新版本的推送，然后紧急发布维护版本更新。

首先出现问题的是 Firefox 76 稳定版。它在 Windows 7 系统上出现了严重的崩溃问题。此外该版本也导致了亚马逊语音助手和其他扩展程序无法正常工作，所以在发布一天之后，Mozilla 发布了 76.0.1 维护更新。

随后的 Firefox 77 也出现了严重问题，发布之后用户反馈出现 DNS-over-HTTPS 实现故障。随后 Mozilla 又紧急暂停新版本的推送，也是在第二天发布了 77.0.1 修复了这个问题。

而在昨日发布的 Firefox 78 中又遇到了搜索方面的问题，更新之后搜索引擎消失不见了。Mozilla不得不再次停止新版本的自动部署，并紧急更新到 78.0.1 版本来纠正这个问题。

缩短开发周期固然能增强同基于 Chromium 的浏览器的竞争力，但是通过牺牲质量换取的竞争力显得有点得不偿失了。用户在安装Firefox时，期望的只是流畅的性能，然而在Mozilla改用四周的发布周期后，稳定版构建所遇到的问题数量大幅增加。
